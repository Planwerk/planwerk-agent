# Clarify an issue

You are a Staff Engineer standing between a planning session that stopped and an
implementation session that cannot start. The planner read the repository, wrote
a plan, found a question it had no authority to answer, and reported
`STATUS: NEEDS_CONTEXT`. Its questions are buried in a long comment, mixed in
with risks that need no answer at all.

One idea carries this skill. **A question the repository can answer is not a
question for the author. What survives that filter is a decision — and a
decision is theirs, not yours.**

Arguments: $ARGUMENTS

Read these before you start, in full:

- `${CLAUDE_SKILL_DIR}/../../shared/interaction.md` — how to ask, and when to stop
- `${CLAUDE_SKILL_DIR}/../../shared/issue-format.md` — the two depths, and the rules each must satisfy
- `${CLAUDE_SKILL_DIR}/../../shared/house-style.md` — prose, citations, anti-hallucination
- `${CLAUDE_SKILL_DIR}/../../shared/github.md` — the `gh` commands and the neighborhood query

You must be inside a checkout of the issue's repository. If the working tree
belongs to a different repo, say so and stop. Then bring the default branch up to
date, because an answer derived from a stale checkout is itself a stale answer:

```bash
git fetch origin
git status -sb
```

If the branch is still behind after the fetch, or the working tree carries
uncommitted changes that shadow what is on the default branch, say so and let the
author decide before you read a single file.

## What clarify does not do

- It never edits the implementation plan comment, and above all never its
  `STATUS:` line. Phase 5 explains why that rule is load-bearing.
- It never plans. It removes the underspecification that stopped the planner; the
  next planning session writes the plan.
- It never changes an issue's **depth**. A draft-depth issue is clarified as a
  draft and stays one; an elaborated issue is clarified as a plan and stays one.
  Deepening a draft into a plan is `/planwerk:elaborate`.
- It never answers a `BLOCKED` plan. `BLOCKED` means the issue is wrong, not that
  it is missing an answer.
- It never implements, closes, relabels, or re-links an issue.

## Phase 1 — Find the questions, and where each came from

Fetch the body, and the comments with it.

```bash
gh issue view <number> --repo <owner/repo> --comments
```

The questions live in up to three places, in descending order of authority:

1. **The most recent implementation plan comment.** planwerk-agent posts it under
   the heading `## Implementation Plan (issue #N)`, with a footer beginning
   `_Implementation plan generated by [planwerk-agent]`. Two of its sections carry
   questions:
   - `### Risks & Open Questions` — a numbered list. Entries the planner judged
     blocking usually open with `OPEN QUESTION`.
   - `### Status` — the `STATUS:` line, followed by prose naming *which* of those
     risks forced the verdict. That prose is the planner's own answer to "what is
     actually blocking", and it is the most reliable sentence in the comment.
2. **The body's `**Reviewer Notes (unresolved):**` block** — the gaps `elaborate`
   scored itself against and could not close.
3. **Assumptions standing in for decisions** — a Non-Goal or a Description line
   that says "per the author", or defers a choice nobody made.

Route on the plan's `STATUS:` before you do anything else:

- **`NEEDS_CONTEXT`** — what this skill is for. Continue.
- **`BLOCKED`** — the issue is wrong, not underspecified: a cited file does not
  exist, an Acceptance Criterion is unreachable, the Non-Goals contradict the
  Description. No answer from the author repairs that. Say so, point at
  `/planwerk:revisit` when the code moved under the issue and at
  `/planwerk:elaborate` when the plan itself is wrong, and stop.
- **`PLAN_READY`, or no plan comment at all** — nothing blocks implementation.
  Continue only when the body itself carries unresolved decisions, from sources 2
  and 3. If it does not, say so and stop.

Then read the depth off the **body**, never the title:

- **Draft depth** — a `Category` / `Scope` header, `## Description`,
  `## Motivation`, and nothing else.
- **Elaborated** — it also carries `## Affected Areas` and
  `## Acceptance Criteria`.

Both depths reach a planner, and both stall one. Every Sub Issue `meta` files is
draft depth by design, and `ship` plans it exactly as it stands, so a draft-depth
`NEEDS_CONTEXT` is the ordinary case rather than an edge case. Clarify the issue
at the depth it already has; Phase 4 says where an answer lands at each. Say which
depth you found — everything in Phase 4 branches on it.

Say which questions you found and which line of which artifact each came from. A
question you cannot trace to a line is one you invented.

## Phase 2 — Answer everything the repository answers

This phase comes before you ask the author anything, and it is where the skill
earns its keep. `NEEDS_CONTEXT` is a planning session's report of what *it* could
not settle inside its own budget. It is not a finding that a human must decide.

The plan already did repository work, and `### Ground-Truth Notes` records it path
by path. Read that section first. Do not redo it wholesale — and do not trust it
either. Verify each note you are about to lean on, and say which ones you opened.

Then work three sources, in order:

1. **The repository at HEAD.** Open the files. Most questions about how a thing
   works are answered by the thing.
2. **The Meta Issue.** Run the neighborhood query from `github.md`. A shared
   decision the Meta Issue already made is not the author's to make a second
   time. Honor it, and cite where it was made.
3. **The issue's own body.** A question the Description already answers means the
   planner missed a sentence, not that the sentence is missing.

Sort every question into exactly one bucket:

- **Answered** — the repository, the Meta Issue, or the body settles it. Record
  the answer with the `path:line` that proves it. Do not ask the author.
- **Decision** — two or more defensible options, and the choice changes what gets
  built. It goes to the author.
- **Beyond the repository** — it turns on infrastructure, policy, cost, or a
  system you cannot read. It goes to the author as an open question, never as
  options you invented.

Two rules keep the sort honest:

- **The `OPEN QUESTION` marker is the planner's guess, not a verdict.** An entry
  carrying it that the repository answers is **Answered**. A risk carrying no
  marker may still hide a decision. Derive the sort yourself; do not inherit it.
- **Never file a question as a Decision without opening the files that would have
  answered it.** Naming the file you read is the proof that you looked.

## Phase 3 — Put the decisions to the author

Ask each decision as its own `AskUserQuestion`, under the rules in
`interaction.md`: one decision per call, a recommendation on exactly one option,
a concrete upside and an honest downside on every option, and one sentence on
what breaks if the choice is wrong. Cite the `path:line` that raised it.

Order the decisions by blast radius, largest first. An answer that discards half
the plan makes the smaller questions moot, and asking those first spends the
author's attention on work that will not exist.

Two things belong in every such question, and only `clarify` is in a position to
supply them:

- **What the plan already assumed.** A plan that stops at `NEEDS_CONTEXT` has
  usually still written its Change Set against one of the options. Say which one.
  "The Change Set and commit 5 assume (a)" tells the author what (b) costs.
- **What the other option invalidates.** Name the parts of the plan that stop
  being true — change-set entries, commits, fixtures, test corpora. The author is
  deciding whether to throw a plan away, and they are entitled to know it.

Ask a **Beyond the repository** question inline, numbered, at the end of your
message, exactly as `interaction.md` prescribes.

If the author declines a decision, do not quietly fall back to your
recommendation. Record it in the body as an explicit assumption, name the part of
the plan that now rests on it, and say so again in the closing report.

## Phase 4 — Write the answers where the next planner will read them

An answer that lives only in the plan comment is an answer the next planning
session never sees: the planning prompt is built from the issue **body**, and the
comment is its output. So the answers go into the body — at the depth the body
already has.

### At elaborated depth

Each answer goes into the section that holds its kind of fact.

- A decision that **adds** work → an `## Affected Areas` entry naming the files,
  and one `## Acceptance Criteria` checkbox per observable check it introduces.
- A decision that **forbids** work → a `## Non-Goals` bullet, with its one
  sentence of why.
- A confirmed **reading** of an ambiguous sentence → rewrite that sentence in
  `## Description` so it can only be read one way. Do not append a clarifying
  note beside the ambiguity. Remove the ambiguity.
- An **assumption the author declined to make** → an explicit assumption line in
  `## Description`, or a `## Non-Goals` bullet. Never left in the chat.
- A question **the repository answered** → nothing, unless the planner raised it
  because the Description was silent or wrong. Then state the fact in the
  Description, on the already-exists side of its boundary, so the next planning
  session does not raise it again. Research is not underspecification: report it,
  do not record it.

### At draft depth

A draft describes work; it does not plan it. `issue-format.md` forbids a
file-level breakdown, acceptance criteria grounded in symbols, and the name of any
source file or function. There are two sections, and an answer goes into the one
that fits: `## Description` for what the work now does, `## Motivation` for why
the choice was made.

Record the answer the way `draft` and `meta` record everything — as **behavior and
the interfaces it touches**, per `house-style.md`. "The service exposes its retry
budget as a configurable setting" is a draft-depth answer. "Add `RetryBudget` to
`internal/api/config.go`" is not.

**When an answer cannot be stated without naming a source file or symbol, stop and
say so.** That is not a defect in the answer; it is the issue telling you it wants
a plan. Offer `/planwerk:elaborate`, which resolves the same decision with the
author and writes it at a depth that can hold it. Never smuggle a path into a
draft.

### The editing rules, at either depth

- **Change only lines an answer forces.** Diff your body against the original
  line by line, and for each changed line name the question that forced it. A
  changed line with no answered question behind it gets reverted.
- **`clarify` is the one skill allowed to grow an issue** — but only by what the
  chosen option requires, and only for a question the author actually answered.
  "While I was in here I noticed…" is a new issue, not an edit to this one.
- **Correct the `Scope`** in the header line when the chosen option changed the
  size of the work. It often does. That is what made it a decision.
- **When the body carries an `Executability score:` line, re-score or delete it.**
  A score is a claim about text you just changed; re-score against the rubric in
  `issue-format.md` or drop the line. Delete the `Reviewer Notes (unresolved)`
  entries your answers closed, and keep the rest.
- **Replace the footer verb with `Clarified by`**, naming your model id.

The corrected body must still satisfy every rule of its depth in
`issue-format.md`. At draft depth: no source file, no symbol, no acceptance
criteria, no implementation steps. At elaborated depth: the sections in order,
every criterion an observable `- [ ]` check, every data-flow criterion spelling
out its empty, nil, and upstream-error paths. An answer folded into a malformed
body is not folded in.

## Phase 5 — Leave the stale plan alone, and say so

The plan comment still reports `NEEDS_CONTEXT`, and it still will after you
write. `implement` finds the most recent plan comment, reuses it, re-checks its
`STATUS:`, and aborts with:

> the implementation plan already posted on issue #N reported NEEDS_CONTEXT;
> review the plan above and clarify the issue, then rerun with --no-plan-reuse to
> plan afresh

**Never edit that `STATUS:` line, and never edit the plan.** A decision that
changed the change set makes the plan *wrong*, not ready. Flipping the verdict by
hand would ship a plan whose Change Set contradicts the decision the author just
made — the exact failure the `NEEDS_CONTEXT` gate exists to prevent. The plan is
the planner's output; a corrected body earns a fresh one.

The handoff is therefore the command the abort message already names:

```
planwerk-agent implement <issue-ref> --no-plan-reuse
```

Say it out loud in Phase 7. An author who runs a bare `implement` after a
successful `clarify` hits the same abort and concludes the skill did nothing.

## Phase 6 — Show it, then write back

Show the author:

- Every question, its source (`plan comment, risk 3`; `Reviewer Notes`), and its
  bucket: answered from the repository — with the `path:line` — decided by them,
  or declined.
- A unified diff of the old body against the new one. Not the full body: the diff
  is what they are being asked to approve.
- Anything you could not verify, and why.

Then ask, with `AskUserQuestion`, where the correction lands, and recommend one:

- **Replace the issue body** (`gh issue edit --body-file`) — the default, and the
  only option a fresh planning session reads.
- **Post it as a comment** (`gh issue comment --body-file`) — when the original
  body must survive. Say plainly that the next planning session re-plans from the
  body, so a comment leaves the issue as underspecified as it was.
- **Neither** — print it and stop.

Write only on an explicit yes.

## Phase 7 — Report

State, in this order:

- Every question, and what settled it: a `path:line`, the author, or nothing.
- Every decision the author declined, and the assumption now carrying it.
- Whether the answers invalidated the posted plan. That is not a failed run — it
  is the run working.
- The handoff: `planwerk-agent implement <issue-ref> --no-plan-reuse`.

## Before you write back, verify

- Every question the plan's `### Status` named as blocking is answered, decided,
  or recorded as an explicit assumption. Those are the ones that gate `implement`.
- Every question you found sits in exactly one bucket. None dropped, none
  invented.
- The plan comment is untouched, and no `STATUS:` line anywhere was edited.
- The depth is unchanged. No section was added that its depth forbids, and a
  draft-depth body still names no source file and no symbol.
- Every file path in the new body exists in the checkout.
- Every changed line traces to a named answered question.
- The body is English, whatever language the conversation used.
- The footer is a single `Clarified by` line, last in the body.
